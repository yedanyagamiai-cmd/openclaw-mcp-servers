ERROR HANDLING APPROACHES
================================================================================

All workers implement these error levels:

1. HTTP-level errors:
   - 400: Invalid Content-Type or malformed JSON
   - 402: Rate limit exceeded (with x402 payment headers)
   - 404: Unknown path (+ honeypot paths)
   - 413: Payload too large (51200-102400 bytes limit)
   - 503: Daily capacity reached (finopsTrack stop threshold)

2. JSON-RPC errors:
   - -32700: Parse error (malformed JSON body)
   - -32600: Invalid JSON-RPC request (missing jsonrpc/method)
   - -32601: Method not found
   - -32029: Rate limit exceeded (in JSON-RPC error response)
   - -32000: Rate limited (prompt-enhancer named constant RATE_LIMITED)
   - -32001: Auth error (prompt-enhancer named constant AUTH_ERROR)

3. Tool-level errors:
   - isError: true in content[] with descriptive message
   - Available tools listed in error for unknown tool name
   - Structured { error: msg, available: [...] } JSON responses

4. Fallback behavior:
   - KV unavailable -> memoryRateLimit (5 req/60s in-memory Map)
   - DB unavailable -> memoryRateLimit fallback
   - All try/catch blocks have silent fallback to allow (fail-open for availability)

KEY OBSERVATIONS & NOTES
================================================================================

1. HIGH CODE REUSE: ~40-50% of each worker is identical boilerplate (shared utility
   functions: memoryRateLimit, validateProKey, proKeyRateLimit, edgeDefense,
   finopsTrack, trackRef, addUpgradePrompt, sanitizeInput, sha256Short, ECOSYSTEM,
   landing HTML, /llms.txt handler). This suggests a shared template/generator.

2. MONETIZATION ARCHITECTURE: The ecosystem uses a freemium model where:
   - "Utility" workers (json, regex, color, timestamp) are fully free with generous limits
   - "Intelligence" workers (intel, fortune, agentforge, moltbook) have pro-gated features
   - All workers share the same Pro key infrastructure (prokey:{key} in KV)
   - A single $9 Pro key works across the entire ecosystem

3. x402 PROTOCOL: All workers implement x402 micro-payment headers on rate limit
   responses, enabling machine-to-machine payment for API access at $0.05/call USDC.
   The payment address (0x72aa56...) is the same across all workers.

4. DETERMINISTIC FORTUNE: The fortune worker's seeded RNG ensures all users get
   identical fortunes for a given date (no per-request randomness), making it
   cacheable and reproducible - useful for testing and consistent UX.

5. STATIC DATA RISK: agentforge-compare stores AI tool data (star counts, versions)
   as static JavaScript objects with a "last_updated: 2026-02-28" timestamp.
   This data will become stale over time without code updates.

6. INTEL WORKER UNIQUENESS: openclaw-intel-mcp is the only worker at version 3.0.0
   (all others are 1.0.0 or 2.0.1 for fortune). It is also the most complex,
   with D1 database reports collected by an external agent.

7. MOLTBOOK PRICING DISCREPANCY: moltbook charges $12 vs $9 for other workers,
   but also accepts the ecosystem $9 Pro key. The $12 appears to be a moltbook-
   specific standalone price for users who only want that tool.

8. JSON-TOOLKIT NO KV: json-toolkit-mcp does not use KV or D1 - it is the only
   fully stateless worker relying entirely on in-memory rate limiting. This means
   rate limits reset on worker restart/cold start.

9. SECURITY CONSISTENCY: All workers implement the same honeypot path list and
   sha256Short IP hashing. The edgeDefense score system (100 starting score,
   -30 for honeypot, -10 for suspicious UA, block at <10, throttle at <30)
   is shared across workers that have KV bound.

10. LLM DISCOVERABILITY: All workers expose /llms.txt and /.well-known/llms.txt
    with standardized discovery format, making them easy to index by LLM directories.
